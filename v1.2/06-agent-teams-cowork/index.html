    06. 에이전트 팀과 코워크 (2026년 2월 업데이트)¶

    Claude Code와 Claude Desktop의 최신 협업 기능을 실무 관점에서 정리했습니다.

    이번 버전에서 달라진 점¶

    2026년 1월~2월 사이에 협업 기능이 빠르게 늘었습니다.

    날짜 변경 내용 실무 영향
    2026-01-12 코워크 연구 미리보기가 Max 플랜(Claude Desktop, macOS)에서 시작 긴 작업을 백그라운드로 돌리는 방식 도입
    2026-01-16 코워크가 Pro 플랜으로 확대 개인 사용자의 접근성 증가
    2026-01-23 코워크가 Team/Enterprise로 확대 팀 단위 운영 가능
    2026년 2월 초 오푸스 4.6 공개 + 에이전트 팀 공개(실험 기능) 병렬 작업 품질과 작업 안정성 개선

    에이전트 팀이란?¶

    에이전트 팀은 Claude Code에서 에이전트 여럿을 한 팀으로 묶어서 동시에 돌리는 기능입니다. 아직 실험 기능이에요.

    서브에이전트 vs 에이전트 팀¶

    둘 다 "에이전트 여럿이 일한다"는 건 같은데, 방식이 완전히 다릅니다.

    항목 서브에이전트 에이전트 팀
    컨텍스트 결과만 호출자에게 반환 각 팀원이 완전히 독립적 컨텍스트 보유
    통신 메인 에이전트에게만 보고 팀원 간 직접 메시지 교환
    조율 메인 에이전트가 중앙 관리 공유 작업 목록으로 자체 조율
    토큰 비용 낮음 높음 (각 팀원이 별도 인스턴스)
    적합한 작업 단일 결과물 위임 역할별 병렬 분해

    활성화 방법¶

    ~/.claude/settings.json 또는 프로젝트별 .claude/settings.json에 추가:

    {
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}

    언제 쓰면 좋은가¶

    • 한 작업을 역할별로 병렬 분해해야 할 때
    • 코드 수정 + 테스트 + 문서화처럼 결과물이 여러 종류일 때
    • 경쟁 가설을 병렬로 테스트할 때 (디버깅)
    • 프론트엔드/백엔드/테스트를 각 팀원이 담당할 때

    실사용 예시: 결제 모듈 리팩토링¶

    목표: 결제 모듈 리팩토링
팀 구성:
- 분석 담당: 현재 결제 흐름과 리스크 식별
- 구현 담당: 결제 로직 분리 및 인터페이스 정리
- 검증 담당: 회귀 테스트 케이스 추가

출력 형식:
1) 변경 요약
2) 파일별 수정 목록
3) 테스트 결과
4) 남은 위험 항목

    표시 모드¶

    모드 설명 팀원 전환
    In-process 메인 터미널 내에서 실행 Shift+Up/Down
    분할 창 각 팀원이 자신의 창 보유 tmux / iTerm2 필요

    팀 운영 팁¶

    • 계획 승인 요구: 팀원이 구현 전에 리더 승인을 받도록 설정 가능
    • 위임 모드: Shift+Tab으로 리더를 조율 전용으로 제한
    • 작업 할당: 리더 직접 할당 또는 팀원 자체 청구 (권장: 팀원당 5~6개 작업)
    • 동일 파일 주의: 여러 팀원이 같은 파일을 동시에 편집하면 충돌 발생

    제한 사항¶

    • 세션 재개 시 in-process 팀원 복원 불가
    • 세션당 하나의 팀만 관리 가능
    • 중첩 팀 불가 (팀원이 자체 팀 생성 불가)
    • 리더 고정 (리더십 이전 불가)

    토큰 비용: 솔직한 이야기¶

    에이전트 팀은 좋긴 한데 토큰을 많이 먹습니다.

    왜 비싼가?¶

    [서브에이전트]
메인 에이전트 1개 + 서브에이전트(결과만 반환)
→ 컨텍스트 공유, 토큰 절약

[에이전트 팀]
리더 1개 + 팀원 3개 = 4개의 독립 인스턴스
→ 각자 별도 컨텍스트 → 토큰 4배
→ 팀원 간 메시지 교환 → 추가 토큰
→ 작업 목록 동기화 → 추가 토큰

    실제 비용 감각¶

    작업 규모 서브에이전트 에이전트 팀 (3인)
    간단한 리팩토링 ~50K 토큰 ~200K 토큰
    모듈 개발 ~100K 토큰 ~400K+ 토큰
    대규모 작업 ~200K 토큰 ~800K+ 토큰

    비용 최적화 전략¶

    1. 연구/검토 작업부터 시작 (파일 수정 없이 분석만)
2. 작업 범위를 명확히 한정 (불필요한 탐색 방지)
3. 팀 규모를 최소로 유지 (2~3명이면 충분)
4. 간단한 작업은 서브에이전트로 (팀이 항상 정답은 아님)

    원칙: 작업이 정말로 독립적이고 병렬 처리가 필요한 경우에만 에이전트 팀을 사용하세요. 순차적 작업이라면 서브에이전트가 더 효율적입니다.

    코워크란?¶

    코워크는 Claude Desktop에서 대화와 별개로 긴 작업을 백그라운드로 돌려놓는 기능입니다. 코딩뿐 아니라 분석, 정리, 자료 작성 같은 일반 업무까지 됩니다.

    현재 지원 범위 (2026년 2월 기준)¶

    • 요금제: Max, Pro, Team, Enterprise
    • 환경: Claude Desktop의 macOS 환경에서 시작
    • 동작 특성: 작업 진행 중에도 다른 일을 할 수 있고, 완료 시 결과를 다시 확인

    시작 순서¶

    1. Claude Desktop에서 모드 선택 메뉴로 이동
    2. 코워크 모드 선택
    3. 큰 작업을 한 번에 넘기되, 출력 형식까지 명시
    4. 완료 후 결과 검수 및 후속 수정 지시

    코워크 요청 템플릿¶

    목표:
- 이번 주 매출/비용/이슈를 종합해 주간 운영 보고서 작성

입력 자료:
- sales.xlsx
- support_tickets.csv
- release_notes.md

반드시 포함:
1) 핵심 지표 5개
2) 전주 대비 변화 원인
3) 다음 주 액션 아이템 3개
4) 경영진용 5문장 요약

    코워크 실전 예시: 이 가이드가 만들어진 과정¶

    이 온보딩 가이드 자체가 코워크의 산물입니다. 실제 과정을 공유합니다.

    1단계: 목적과 대상 정의¶

    > ai 관련 글을 쓰려고.
> 내가 어떻게 쓰는지를 이야기하고 싶어.
> 대상은 우리 개발 부서에 대해서야.
> 기본적으로 claude 사용법 + claude code 사용법
> 그리고 ai에 대한 내용을 적어보고자 해.

    처음부터 완벽한 구조를 잡으려 하지 않았어요. "이런 글을 쓰고 싶다"는 방향만 던졌습니다.

    2단계: AI가 자료를 분석하고 구조 제안¶

    Claude가 제공된 자료들(.claude 폴더 설정, 참조 글, 에이전트 파일)을 모두 읽고 발표 전략을 제안했습니다:

    Claude: "참조 글의 핵심 원리인 '인지부하 최적화'를
        발표 자체에도 적용하면 됩니다.

        발표 구조 제안:
        1단계: 충격/공감 → '어제 버그 잡는데 몇 시간 걸렸나요?'
        2단계: 문제 정의 → Copy Loop vs Verification Loop
        3단계: 개념 설치 → Agent, Rules, Hook, Permission
        4단계: 시연 → 실제 .claude/ 설정으로 버그 해결
        5단계: 연결 → '내일부터 이렇게 써보세요'"

    3단계: 대화를 통한 발전¶

    한 번에 완성된 게 아닙니다. 대화를 거듭하면서 구조가 잡혔어요:

    > react-patterns.md를 어떻게 활용할 수 있을까?

Claude: "발표에서 '팀의 스키마'로 사용하세요.
        이 규칙이 AI 머릿속에 들어있다는 걸 보여주면
        청자가 '아, 이렇게 설정하면 되는구나' 이해합니다."

> 지루하지 않게 만들려면?

Claude: "라이브 데모가 설명을 이깁니다.
        실제 버그를 Claude Code로 해결하는 과정,
        hook이 console.log를 잡는 순간을 보여주세요."

    4단계: 최종 산출물¶

    이 대화 과정이 바로 이 온보딩 가이드의 기반이 되었습니다.

    핵심 교훈:

    • 처음부터 완벽할 필요 없음 → 방향만 잡고 시작
    • AI가 자료 분석하고 구조 제안 → 선택은 내가
    • 대화하면서 점점 나아짐 → 갈군다
    • 실제 설정 파일이 가장 좋은 예시 → 보여주기

    오푸스 4.6과 코워크¶

    오푸스 4.6은 에이전트형 코딩 안정성, 긴 추론, 어려운 작업의 품질이 좋아졌습니다. 파워포인트 애드인에서는 소넷 4.5와 오푸스 4.6을 골라 쓸 수 있어요.

    주의:

    • 공식 문서에 "오푸스 4.6이 코워크 전용 기능을 직접 추가했다"는 문구가 명시된 것은 아닙니다.
    • 다만 모델 성능 향상과 애드인 모델 선택 확장을 고려하면, 코워크 결과물 품질 향상에 유리하다고 해석할 수 있습니다.

    운영 팁:

    • 탐색/초안은 소넷 계열로 시작하고
    • 최종 검토/난도 높은 판단은 오푸스 4.6으로 마감
    • 결과물 제출 전 반드시 사람이 숫자/출처/정책 준수 여부를 검수

    엑셀에서 Claude: 실전 가이드¶

    1) 설치¶

    • 개인 사용자: 오피스 추가 기능 스토어에서 Claude 추가 기능 설치
    • 조직 사용자: 마이크로소프트 365 관리 센터에서 관리자가 조직 배포

    2) 작업 방식¶

    워크북 안에서 Claude 패널을 열고 범위를 지정해 요청합니다.

    추천 요청 패턴¶

    이 시트의 B열~F열을 기준으로
1) 누락값과 이상치를 먼저 찾고
2) 월별/카테고리별 집계를 만든 뒤
3) 경영진이 읽을 7줄 요약을 작성해줘.
수식은 왜 그렇게 만들었는지 한 줄 설명도 포함해줘.

    이 데이터로 피벗 테이블 2개를 제안해줘.
- 하나는 매출 추세
- 하나는 지역별 수익성
각 피벗에서 봐야 할 핵심 해석 포인트를 3개씩 적어줘.

    3) 결과 검수 체크리스트¶

    • 통화/날짜 형식이 원본 규칙과 같은가?
    • 분모가 0인 계산이 안전하게 처리됐는가?
    • 피벗/차트 범위에 누락 행이 없는가?
    • 요약 문장이 실제 수치와 일치하는가?

    4) 버전 참고¶

    엑셀 도움말 문서의 모델 표기는 시점에 따라 빠르게 바뀝니다. 문서와 실제 화면의 모델 선택 항목이 다르면 화면 기준으로 우선 확인하세요.

    파워포인트에서 Claude: 실전 가이드¶

    1) 설치¶

    • 개인 사용자: 오피스 추가 기능 스토어에서 설치
    • 조직 사용자: 관리자 일괄 배포 권장

    2) 슬라이드 작성 흐름¶

    1. 발표 목적/청중/시간(예: 10분)을 먼저 전달
    2. 슬라이드 목차 초안 생성
    3. 슬라이드별 핵심 메시지 1개씩 고정
    4. 마지막에 톤/길이/중복만 정리

    추천 요청 패턴¶

    대상: 임원진
목적: 2026년 1분기 운영 성과 보고
분량: 10장 이내

요청:
1) 장표 구조(제목 + 핵심 메시지) 제안
2) 각 장표의 말할 거리 3개
3) 마지막 장표에 의사결정 요청 항목 2개 추가

    3) 모델 운영 팁¶

    • 빠른 수정 반복: 소넷 계열
    • 메시지 정제/고난도 서술: 오푸스 4.6
    • 최종본은 발표자가 직접 읽어보며 문장 길이와 용어 통일 점검

    팀 적용 체크리스트¶
